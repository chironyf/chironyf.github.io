数据库课设之宾馆管理系统

Jan 11, 2018 | 阅读

从课设选题，到前期需求分析，再到编码实现以及后期测试和优化等，前前后后差不多弄了3个星期的时间（三人团队开发），这套管理系统已经具备订房退房、续费、房间管理、订单管理、员工管理、业务数据可视化展现等功能。

项目结构

系统分为模型层和视图层，关于视图层需要说明一下，这与传统的MVC架构并不一样，因为控制器并没有独立出来，大部分与视图混在一起，姑且就叫做视图层吧，这样做虽然降低了各个模块之间的独立性，但是加快了开发速度，不同的成员根据自己的功能去创建对应的视图页面，在模型层的基础上进行简单的数据控制即可完成对应功能的实现。

  • 模型层（后端，java包）主要用于数据库的连接以及相关的数据提取
    • config
      全局配置文件，比如数据库连接信息，字符串常量等
    • display
      视图对象实体，从基本表实体抽象出来
    • entity
      基本表实体
    • servlet
      处理简单的页面请求
    • tool
      用于连接数据库以及相关数据的提取
  • 视图层（前端，webapp包）用于生成前端页面以及控制数据如何展示，按项目功能分为
    • orderManagement
      订单管理（这个包下面的jsp页面基本是重复的，由于时间关系，并没有简化）
    • roomManagement
      客房管理，添加房间、房间展示、房间搜索
    • systemManagement
      系统管理员登录的页面
    • 根目录下的几个JSP文件
      包括登录界面、两种登录模式的菜单栏、以及订房续费退房等

数据库设计

一共建立了9个基本表、5个视图、8个触发器、5个双列索引和3个单列索引（除去自动生成的主码和外码索引）以及两个授权用户system和hotel，其中system用户拥有对所有表的操作权限，它一个用途是用来验证登录信息是否正确，另一个用途是作为系统管理员登录时访问数据库的用户，宾馆管理员在登录时也是通过system查表验证，为了限制它的权限，登录成功后立马将数据库登录用户切换成hotel。hotel用户只有部分权限，比如它不能对waiter表进行select。
从最初的设计开始到后来定下来，反复修改过好几次，比较坑的是，在InnoDB引擎中并不支持check约束，连简单的性别男女都检查不了，但在JSP页面中可以直接通过下拉选项（性别默认就男女两种）来保证每次插入数据库中的性别，需要注意的是，这并没有体现用户定义的完整性。

总结

  • 前期建表的时候，订单表名用的是order，在执行sql脚本的时候一直报错，而且莫名其妙的错误，找了半天才发现这个与关键字冲突了，后来才加上了s
  • 项目中最先使用的ECharts是官方完整版本中的echarts.js，但是在渐变折线图时发现折线无法显示，由于可能是库的原因，就换了源代码版本的echarts.js，果然就能正常显示了，这个版本也是官方建议开发环境下使用的。
  • 表单验证前期采用的全是alert的方式，在后期界面优化的时候全部换成了semantic-ui中的风格。
  • 订单管理的分页功能是在验收的前一天加上去的，通过url中的pageIndex来控制显示的页码，在下面的搜索框中输入页码即可前往指定的页面，在url和输入框中均做了错误处理，你可以直接在下方的输入框中输入不正确的字符比如”fghjk”或者-1000，但还是能显示正常的页面。
  • 各个控件的图标也是后期加上去的，吐槽一下semantic-ui找图标的功能，真的很鸡肋，连基本的搜索都没有，光找这十几个图标，就花了好几小时。
  • 页面的导航栏是通过include标签引入的，下面的部分放在一个pusher包裹的container中，在一次小组同步代码的时候，莫名其妙的是，导航栏和下面的容器之间就多了一段空白，查了好几遍也没发现问题。
  • 用cloc统计了一下，共45个文件，JSP + Java代码差不多5700行，源码已经托管在GitHub上，项目地址请点这里。
