数据库课设之宾馆管理系统

Jan 11, 2018 | 阅读

从课设选题，到前期需求分析，再到编码实现以及后期测试和优化等，前前后后差不多弄了3个星期的时间，这套管理系统已经具备订房退房、续费、订单管理、员工管理、业务数据可视化展现等功能。

项目结构

分为模型层和视图层，关于视图层需要说明一下，这与传统的MVC架构并不一样，因为控制器并没有独立出来，大部分与视图混在一起，姑且就叫做视图层吧。

  • 模型层（java包）主要用于数据库的连接以及相关的数据提取
    • config
      全局配置文件
    • display
      视图对象实体
    • entity
      基本表实体
    • servlet
      用于处理简单的页面请求（主要就是重定向）。
  • 视图层（webapp包）用于生成前端页面以及控制数据如何展示，按项目功能分为
    • orderManagement
      订单管理（这个包下面的jsp页面基本是重复的，由于时间关系，并没有简化）
    • roomManagement
      客房管理，添加房间、房间展示、房间搜索
    • systemManagement
      系统管理员登录的页面
    • 其他
      根目录下的几个JSP文件，登录界面、两种登录模式的菜单栏、以及订房续费退房。

数据库设计

一共建立了9个基本表、5个视图、8个触发器、5个双列索引和3个单列索引（除去自动生成的主码和外码索引）以及两个授权用户system和hotel，其中system用户拥有对所有表的操作权限，它一个用途是用来验证登录名是否合法，另一个用途是作为系统管理员登录时访问数据库的用户，宾馆管理员在登录时也是通过system查表验证，为了限制它的权限，登录成功后立马将数据库登录用户切换成hotel。hotel用户只有部分权限，比如它不能对waiter表进行select。
从最初的设计开始到后来定下来，反复修改过好几次，比较坑的是，在InnoDB引擎中并不支持check约束，连简单的性别男女都检查不了，但在JSP页面中可以直接通过下拉选项（性别默认就男女两种）来保证每次插入数据库中的性别，需要注意的是，这并不能体现用户自定义完整性。
